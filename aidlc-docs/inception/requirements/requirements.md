# Requirements - Velaris Phase 1 默认路径收束

## 1. 意图分析摘要

- **用户请求**：安装 `aidlc`，并执行上一轮确认方案中的上游 Phase
- **当前执行假设**：这里的“上面的 Phase”解释为 `Phase 1：默认路径收束 / 最小可信闭环`
- **请求类型**：架构增强 + 运行时语义收束 + 文档/实现对齐
- **范围估计**：多组件、多文档、多测试的 Brownfield 变更
- **复杂度估计**：Complex

## 2. 问题背景

用户在 2026-04-18 明确给出的架构口径是：

- `Velaris` 是整个系统架构主体与品牌承载层
- `OpenHarness` 只是 Harness 执行基座
- 后续设计与实现应基于逆向工程信息继续推进，而不是把 `Velaris` 降格为单纯挂载在 `OpenHarness` 上的局部子系统

现有仓库已经形成了：

- `Velaris` 决策架构主体（规划、路由、授权、执行、记忆、学习）
- `OpenHarness` 通用执行基座（会话、工具调度、模型调用、UI/CLI/MCP 等）
- 文件型 `DecisionMemory`
- 可选 PostgreSQL 持久化增强路径

当前主要问题不是“完全没有能力”，而是：

- 默认路径与增强路径语义不等价
- 会话恢复与业务执行恢复边界混杂
- 授权对象已存在，但尚未形成强执行门
- 对外定位、代码组织和默认实现语义之间仍有偏差

## 3. 当前默认需求边界

在未收到问题文件答复前，当前草案把本轮目标限定为：

1. 明确 `Velaris` 的正式定位：整体架构主体与决策内核
2. 明确 `OpenHarness` 的正式定位：执行 Harness 基座，而不是架构主体
3. 明确默认路径下哪些对象是权威状态，哪些只是派生视图
4. 明确“会话”和“业务执行”是两个不同层级
5. 为完成语义与恢复语义建立统一 contract
6. 同步修正文档表述，避免能力叙事超前于默认实现

## 4. Functional Requirements

### FR-1 定位收束

系统必须把 `Velaris` 明确建模为：

- 整体架构主体与决策能力承载层
- 对外产品与架构叙事的第一命名空间
- 负责定义系统中的决策语义、治理语义、恢复语义和学习语义

系统必须把 `OpenHarness` 明确建模为：

- 执行 Harness 基座
- 提供会话、工具调度、推理循环、MCP、UI/CLI 等通用运行能力
- 作为 `Velaris` 的底座，而不是主导业务语义的架构主体

### FR-2 一等执行单元

系统应定义独立于会话层的业务执行单元，用于承载：

- execution id
- 生命周期状态
- 执行上下文
- 结果收口
- 恢复边界

### FR-3 默认状态链说明清晰

系统必须明确以下对象在默认路径中的角色：

- 主状态
- 派生记录
- 重启后必须可恢复的状态
- 可由恢复过程重建的状态

### FR-4 完成 contract

系统必须对以下完成语义给出统一定义：

- 结构完成
- 约束完成
- 目标完成
- 必须失败
- 部分完成

### FR-5 恢复 contract

系统必须显式区分并定义：

- 会话恢复
- 执行恢复
- outcome 恢复
- audit 恢复
- 中断点恢复

### FR-6 授权到执行门

系统应让已有授权对象逐步具备执行含义，包括但不限于：

- 哪一层消费授权结果
- 哪一步执行校验
- 校验失败时如何 fail-closed

### FR-7 文档与实现一致性

系统应同步修正 README、设计文档和 AI-DLC 文档中的定位与边界，确保：

- `Velaris` 被表述为整体决策架构
- `OpenHarness` 被表述为执行基座
- 默认路径能力与增强路径能力被清晰区分

## 5. Non-Functional Requirements

### NFR-1 向后兼容

本轮改动应尽量保持：

- `velaris` / `vl` / `openharness` / `oh` 入口兼容
- 现有业务场景工具名与基础调用方式兼容

### NFR-2 默认路径优先

默认路径必须先形成最小可信闭环；增强路径只能放大已成立语义，不能替默认路径补语义。

### NFR-3 可验证性

本轮输出必须能通过聚焦测试、契约测试或运行验证证明“默认语义比当前更清晰”，而不是只停留在文档层。

### NFR-4 控制复杂度

本轮不应优先引入更多场景名词、治理名词或平台分层；应优先收束现有对象语义。
